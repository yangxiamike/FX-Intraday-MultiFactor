# Git Workflow

版本：`v0.1`

最后更新：`2026-04-03`

## 1. 目的

本文件定义本项目的 git 使用规范。

目标：

- 让每次开发修改都有可追溯记录
- 让代码修改和文档修改保持一致
- 避免“代码改了，文档没跟上”

## 2. 基本原则

- 每次开发前先确认任务范围
- 每次开发后必须更新相关文档
- 每次有效修改后必须做 git 维护
- 提交粒度要围绕单一任务，不要把无关改动混在一起

## 3. 开发后的固定动作

每次完成一组相关修改后，默认执行下面动作：

1. 查看本次修改范围
2. 检查哪些文档需要更新
3. 更新相关文档
4. 再次检查 diff
5. 只暂存本次任务相关文件
6. 创建能准确描述修改内容的提交

## 4. 文档同步规则

常见对应关系：

- 改了项目总览、运行方式、文档入口：更新 `README.md`
- 改了规格、接口、字段语义、验收口径：更新 `spec.md`
- 改了任务顺序、里程碑、阶段目标：更新 `plan.md`
- 改了模块职责、调用关系、关键设计决策：更新 `ARCHITECTURE.md`
- 当前状态、阻塞、下一步变化：更新 `CONTEXT.md`
- 推进了阶段性里程碑：更新 `docs/progress.md`
- 改了整体结构可视化表达：更新 `architecture_mindmap.md`
- 发现新错误、新坑、新经验：更新 `docs/errors_and_lessons.md`
- 改了开发流程或版本管理习惯：更新本文件

## 5. 提交粒度要求

推荐提交粒度：

- 一次提交只覆盖一组相关变更
- 文档和代码如果属于同一个任务，可以放进同一个提交
- 不要把“文档整理”“数据链路”“回测重构”混成一次提交

## 6. 提交说明建议

推荐风格：

- `docs: refine round-1 architecture and workflow`
- `data: add USDJPY file ingestion pipeline`
- `research: add factor evaluation reports`
- `backtest: add vectorized strategy engine`

要求：

- 提交说明要能看出本次修改的主题
- 不要使用过于模糊的说明，如 `update`、`fix stuff`

## 7. 分支建议

建议：

- 主分支用于稳定版本
- 新任务在独立分支开发
- 合并前确认文档已同步

如果当前还是单人早期开发，也至少要保证：

- 提交有清晰主题
- 重要节点可回退

## 8. 当前状态说明

截至当前：

- 项目刚开始建立 git 规范
- 本地 git 仓库已初始化
- 从现在开始，本规则生效

## 9. 后续维护

如果后续下面任一项发生变化，需要更新本文件：

- 提交习惯变化
- 分支策略变化
- 文档同步规则变化
- 发布流程变化
